# Storybook a11y — setup & rationale

Sprint 070 audit (2026-06-01).

## Two layers, complementary not duplicate

### Layer 1 — `@storybook/addon-a11y` (Sprint 067)

- Registered in `main.ts`: `addons: ['@storybook/addon-a11y']`
- Runs in the **Storybook dev UI** (panel below each story)
- Powered by axe-core under the hood
- **Interactive feedback** for developers while authoring stories
- Does NOT enforce in CI by default

### Layer 2 — Test-runner `postVisit` hook (Sprint 069)

- Defined in `test-runner.ts`
- Runs in **CI** via `nx test-storybook core` (Playwright-driven)
- Uses `@axe-core/playwright` (`AxeBuilder` against the live story iframe)
- Filters to `wcag2a` + `wcag2aa` tags, fails on `critical` / `serious` impact
- **Programmatic gate**, complements dev-panel feedback

## Why both?

Sprint 069 PoC raised a question: does Layer 2 duplicate Layer 1?

Answer: **no**. The two layers serve different audiences and contexts:

| Aspect      | addon-a11y (Layer 1)     | postVisit (Layer 2)             |
| ----------- | ------------------------ | ------------------------------- |
| Surface     | Storybook UI panel       | Test-runner CLI output          |
| Trigger     | Dev opens a story        | `nx test-storybook core`        |
| Enforcement | Advisory (visual)        | Blocking (throws on violations) |
| Coverage    | The story you're viewing | All stories in one run          |

Storybook 8+ also ships a Vitest-based `@storybook/experimental-addon-test`
that can run addon-a11y assertions inside Vitest. That is a **third** path,
distinct from the Playwright test-runner. We don't use it (Vitest isn't part
of this stack — Jest powers the unit tests). The Playwright test-runner +
manual `postVisit` remains the documented pattern for axe gating.

Reference: https://storybook.js.org/docs/writing-tests/accessibility-testing

## Per-story disable

If a story has a known violation that's intentional (e.g. demoing a contrast
failure), opt it out:

```ts
export const Demo: Story = {
  parameters: { a11y: { disable: true } },
};
```

The `postVisit` hook respects this:

```ts
if (storyContext.parameters?.['a11y']?.disable) return;
```

## Running the gate locally

```bash
# Terminal 1
pnpm exec nx storybook core
# Wait for "Storybook X.X started" on http://localhost:4400

# Terminal 2
pnpm exec nx test-storybook core
```

Expected runtime (Sprint 070 baseline): ~115 components × ~3 stories each
~ 345 stories × 3–5s/story ≈ **17–29 minutes** for a full broad sweep.
Plan for this duration when scheduling CI gates.
